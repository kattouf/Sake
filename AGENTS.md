# AGENTS.md

## Project Overview

Sake is a Swift-based command runner inspired by Make. Two-part system:
- **sake** (CLI) — locates, compiles, and executes SakeApp
- **SakeApp** — user's SPM executable containing commands defined via `@CommandGroup` macro

Requirements: macOS 13+, Linux, Swift 6.1+.

## Module Architecture

```
SakeCLI            → CLI entry point (@main, ArgumentParser). Thin layer delegating to SakeCLILibrary.
SakeCLILibrary     → Core CLI logic: config resolution, SakeApp build/run/init, shell execution.
Sake               → Public library: Command, CommandGroup, SakeApp protocol, CommandRunner, CLI (embedded in SakeApp binary).
SakeShared         → Shared utilities: CaseConvertingStrategy, ProcessMonitor, SakeAppExitCode, CommandGroupsJSON.
SakeMacros         → Swift macro: @CommandGroup generates CommandGroup conformance + `commands` dict.
```

Dependency flow: `SakeCLI → SakeCLILibrary → SakeShared`, `Sake → SakeShared`, `Sake → SakeMacros`.

## Key Types

- `Command` — Sendable struct: description, dependencies, runDependenciesConcurrently, skipIf closure, run closure. All closures are `@Sendable (Context) async throws`.
- `Command.Context` — arguments, environment, appDirectory, runDirectory, Storage (thread-safe dict with NSRecursiveLock), InterruptionHandler.
- `CommandGroup` protocol — `name: String`, `commands: [String: Command]`. Auto-generated by `@CommandGroup` macro.
- `SakeApp` protocol — extends `CommandGroup`, adds `configuration: SakeAppConfiguration` (lists additional command groups).
- `CommandRunner` — executes command: checks skipIf, runs dependencies (sequential or concurrent via `withThrowingTaskGroup`), then runs main logic.
- `SakeAppManager<Mode>` — phantom types `UninitializedMode`/`InitializedMode` for compile-time state safety. `~Copyable`.

## Command Visibility Rules

Only `public static var` properties of type `Command` or `Sake.Command` are exposed to CLI. Non-public or non-static commands are internal (usable as dependencies but not callable from terminal). Typealiases don't work.

## Configuration

Three-level cascade (highest priority first): CLI args → ENV vars → `.sake.yml` file → defaults.

Key options: `sakeAppPath`, `sakeAppPrebuiltBinaryPath` (mutually exclusive), `caseConvertingStrategy` (keepOriginal/toSnakeCase/toKebabCase).

## Build & Test

```bash
sake build_tests       # Build test targets
sake unit_tests        # Run unit tests (--skip-build to skip build)
sake integration_tests # Run integration tests
sake test              # All of the above
sake format            # SwiftFormat
sake lint              # SwiftFormat --lint
```

CI runs matrix: macOS + Ubuntu, Swift 6.1 + 6.2.1.

## Code Style

- SwiftFormat 0.56.3 (configured in `.swiftformat`)
- Swift version: 6.1
- Max line width: 140
- Wrap arguments/parameters/collections: `before-first`
- Enabled rules: `isEmpty`, `wrapSwitchCases`, `wrapConditionalBodies`, `wrapEnumCases`
- Disabled: `redundantRawValues`, `redundantSelf`, `redundantStaticSelf`, `redundantType`, `preferKeyPath`
- `package` access level used for cross-module internal APIs (SPM package access)

## Git Workflow & Release Notes

Release notes are generated by git-cliff from commit messages on main. Two patterns for landing code:

### Pattern 1: Direct commit to main

For small, low-risk changes (config tweaks, version bumps, single-file fixes).

Commit message **must** have a conventional commit prefix — it goes directly into release notes.

```
fix: binPath location
project: update SakeApp deps
doc: actualize deps versions
```

### Pattern 2: Pull Request (squash merge)

For larger changes. **Only squash merge is allowed** (merge commit and rebase are disabled).

- **PR title must** have a conventional commit prefix — enforced by `semantic-pr-lint.yml` (action-semantic-pull-request). PR title becomes the squash commit message and lands in release notes.
- **Commits inside the branch** — free format, no prefix required (they get squashed away).
- **Branch naming** — kebab-case, optionally prefixed with type (`feat-`, `fix-`, `doc-`). Not enforced, not used for release notes.
- **CI checks on PR** — tests (matrix: macOS + Ubuntu, Swift 6.1 + 6.2.1) and lint must pass.

### Which to use

**For AI agents: always use PR for non-trivial changes.** This ensures CI checks pass before merge. Direct commits to main skip PR checks (only post-push CI runs).

### Conventional Commit Types

Validated by `semantic-pr-lint.yml` and parsed by git-cliff:
- `feat:` — Features
- `fix:` — Bug Fixes
- `test:` — Testing
- `perf:` — Performance
- `refactor:` — Refactoring
- `doc:` — Documentation
- `project:` — Project changes
- `chore:` — Maintenance (with optional scope, e.g. `chore(deps):`)
- `revert:` — Reverts

Note: `chore(release): Bump version to X` commits are skipped in release notes.

## CI Workflows

- **checks.yml** — on push to main and PRs: tests (matrix macOS+Ubuntu, Swift 6.1+6.2.1) + lint
- **semantic-pr-lint.yml** — validates PR title has conventional commit prefix
- **coverage.yml** — on push to main: code coverage via Codecov (macOS-15, Xcode 16.3)
- **docs-deploy.yml** — on push to main (docs/** changes): deploys VitePress docs to GitHub Pages
- **dependabot.yml** — monthly Swift dependency updates with `chore(deps):` prefix
- **setup-environment action** — composite action: mise + Swift + setup-sake

## Testing Patterns

- XCTest framework
- Unit tests: `SakeTests`, `SakeCLILibraryTests`
- Macro tests: `SakeMacrosTests` using `SwiftSyntaxMacrosTestSupport` / `assertMacroExpansion`
- Integration tests: `IntegrationTests` (full CLI execution via subprocess)
- Test helper: `Command.Context.empty` for creating minimal test contexts
- Pattern: `nonisolated(unsafe) var runnedCommands: [String] = []` for tracking execution order

## Error Handling

- `SakeAppError` — command-level errors with `LocalizedError` conformance and helpful messages (closest match suggestions via Levenshtein distance)
- `SakeAppManagerError` — CLI-level errors (init/build/validation failures)
- Custom exit codes in `SakeAppExitCode` (101-105)
